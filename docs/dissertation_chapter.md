# Глава. Проектирование и разработка встроенного приложения Bitrix24 для автоматизации деятельности туристической компании

## Модуль 1. Структура проекта

Назначение модуля структуры проекта состоит в формировании воспроизводимой основы программной системы, пригодной для промышленной эксплуатации и дальнейшего развития. В рамках данного этапа приложение разделено на публичную точку входа, слой маршрутизации, контроллеры, сервисы интеграции, репозитории доступа к данным, представления пользовательского интерфейса, миграции базы данных, тесты и эксплуатационную документацию. Необходимость разработки данной структуры обусловлена тем, что встроенное приложение Bitrix24 должно одновременно решать задачи пользовательского взаимодействия, хранения локальных бизнес-данных и интеграции с внешней REST-платформой, поэтому монолитный процедурный сценарий не обеспечивает требуемой сопровождаемости.

Архитектура реализована в стиле MVC. Контроллер принимает HTTP-запрос, вызывает доменную или интеграционную логику и передает данные в представление. Репозитории инкапсулируют операции MySQL, а сервисы инкапсулируют взаимодействие с Bitrix24. Такой подход соответствует принципу единственной ответственности: маршрутизатор не знает о базе данных, представление не выполняет REST-запросы, а сервис Bitrix24 не содержит HTML-разметку. Алгоритм работы начинается с загрузки автозагрузчика, инициализации контейнера, регистрации маршрутов и передачи управления контроллеру, соответствующему URL.

Структура классов включает `Container`, отвечающий за конфигурацию окружения и создание инфраструктурных зависимостей, `Router`, выполняющий диспетчеризацию HTTP-запросов, базовый `Controller`, базовый `Repository`, ресурсные контроллеры для справочников и сервисы интеграции. Пользовательский интерфейс строится на Bootstrap 5, что обеспечивает единый визуальный язык, адаптивность и пригодность к встраиванию во фрейм Bitrix24. На данном этапе непосредственный обмен с Bitrix24 ограничен подготовкой слоя интеграции, однако структура уже предусматривает загрузку BX24 JavaScript SDK и серверные REST-вызовы.

База данных описана SQL-миграцией, в которой выделены таблицы клиентов, туристических заявок, туров и коммуникаций. Входными данными модуля являются HTTP-запрос, переменные окружения и параметры установки приложения. Выходными данными являются HTML-страницы, записи в MySQL и REST-запросы к Bitrix24. Бизнес-логика данного этапа заключается в обеспечении устойчивого каркаса, на котором строятся функциональные модули. Ошибки централизованно передаются обработчику исключений, что исключает раскрытие технических деталей конечному пользователю.

Основной листинг точки входа показывает, что приложение не смешивает регистрацию маршрутов с реализацией бизнес-операций. Файл `public/index.php` загружает автозагрузчик, создает контейнер и связывает URL с контроллерами. После этого маршрутизатор передает выполнение конкретному обработчику.

```php
$container = Container::boot(dirname(__DIR__));
$router = new Router($container);
$router->get('/', App\Controller\DashboardController::class, 'index');
$router->get('/clients', App\Controller\ClientController::class, 'index');
$router->post('/clients/store', App\Controller\ClientController::class, 'store');
$router->dispatch($_SERVER['REQUEST_METHOD'], parse_url($_SERVER['REQUEST_URI'], PHP_URL_PATH) ?: '/');
```

## Модуль 2. Настройка Embedded Application

Назначение модуля настройки Embedded Application состоит в обеспечении корректного жизненного цикла приложения внутри портала Bitrix24. Встроенное приложение должно иметь отдельный URL установки, URL размещения и возможность корректно инициализироваться внутри интерфейса портала. Необходимость данного модуля определяется спецификой Bitrix24: пользователь не открывает приложение как независимый сайт, а запускает его из интерфейса портала, где действуют правила авторизации, ограничения iframe и механизм передачи параметров установки.

Архитектура модуля разделяет серверный этап установки и клиентский этап размещения. Серверный сценарий `install.php` принимает параметры, переданные Bitrix24, и сохраняет учетные данные. Клиентский сценарий `placement.php` подключает JavaScript SDK и после `BX24.init` переводит пользователя в основное приложение. Алгоритм работы заключается в том, что Bitrix24 вызывает install URL, передает `AUTH_ID`, `REFRESH_ID`, `DOMAIN`, `member_id` и `client_endpoint`, после чего приложение фиксирует эти сведения и становится способным выполнять REST-запросы от имени установленного приложения.

Пользовательский интерфейс на данном этапе минимален, поскольку основная цель состоит не в отображении данных, а в корректном подключении к среде Bitrix24. Взаимодействие с Bitrix24 выполняется через параметры установки и BX24 JavaScript SDK. REST-методы на этапе установки напрямую не вызываются, однако сохраняется `client_endpoint`, который в дальнейшем используется REST-клиентом. Входными данными являются параметры install callback, выходными данными является сохраненное состояние авторизации и перенаправление в приложение.

Листинг установки демонстрирует минимальную серверную точку входа, которая делегирует всю обработку сервису авторизации. Это снижает связанность и позволяет тестировать обработку токенов отдельно от HTTP-оболочки.

```php
$auth = new App\Service\BitrixAuthService(new App\Service\TokenStorageService());
$auth->handleInstall($_REQUEST);
header('Location: /?installed=1');
```

## Модуль 3. Авторизация через BX24

Модуль авторизации предназначен для сохранения и последующего использования токенов доступа, полученных от Bitrix24. Его необходимость обусловлена тем, что серверное приложение не может выполнять REST-вызовы без `AUTH_ID`, а также должно знать домен портала и endpoint REST API. Архитектурно модуль представлен сервисами `BitrixAuthService` и `TokenStorageService`. Первый интерпретирует параметры установки, второй отвечает за персистентное хранение учетных данных.

Алгоритм работы начинается с проверки наличия обязательных параметров `AUTH_ID` и `DOMAIN`. Если они присутствуют, формируется структура токена, содержащая домен, access token, refresh token, client endpoint и member id. Затем структура сериализуется в JSON-файл в каталоге `storage`. В промышленном варианте данный механизм может быть заменен на хранение в MySQL с шифрованием секретов, однако интерфейс сервиса останется неизменным.

Взаимодействие с Bitrix24 происходит на уровне протокола установки. REST-методы на этом этапе не вызываются, но результат авторизации становится входным условием для всех последующих REST-операций. Входными данными являются параметры установки, выходными данными является сохраненный токен. Ошибочная ситуация отсутствия обязательных параметров обрабатывается безопасно: сервис прекращает выполнение без записи некорректного состояния.

```php
$this->storage->save([
    'domain' => $request['DOMAIN'],
    'access_token' => $request['AUTH_ID'],
    'refresh_token' => $request['REFRESH_ID'] ?? '',
    'client_endpoint' => $request['client_endpoint'] ?? ('https://' . $request['DOMAIN'] . '/rest/'),
    'member_id' => $request['member_id'] ?? '',
]);
```

## Модуль 4. Подключение REST API Bitrix24

Назначение REST-модуля состоит в унификации серверного обмена с Bitrix24. Необходимость такого слоя связана с тем, что различные бизнес-модули используют разные REST-методы, но механизм вызова одинаков: необходимо получить endpoint, добавить токен авторизации, отправить POST-запрос и декодировать JSON-ответ. Архитектура модуля представлена классом `BitrixApiClient`, который зависит от хранилища токенов и предоставляет единый метод `call`.

Алгоритм работы REST-клиента включает получение текущего токена, проверку его наличия, построение URL вида `client_endpoint/method.json`, формирование тела `application/x-www-form-urlencoded`, добавление параметра `auth` и преобразование ответа Bitrix24 в массив PHP. Используемые REST-методы зависят от вызывающего сервиса. В текущей реализации применяются `profile` для проверки контекста пользователя, `im.notify.system.add` для уведомлений и `tasks.task.add` для создания задач. Входными данными являются имя REST-метода и массив параметров, выходными данными является массив результата Bitrix24.

Обработка ошибок реализована через безопасный результат `NO_TOKEN`, если приложение еще не установлено. Сетевые ошибки в дальнейшей промышленной версии целесообразно оборачивать в доменные исключения и фиксировать в журнале, однако уже сейчас вызов REST изолирован от контроллеров, что позволяет централизованно расширить обработку ошибок.

```php
$url = rtrim($token['client_endpoint'], '/') . '/' . $method . '.json';
$payload = http_build_query(array_merge($params, ['auth' => $token['access_token']]));
$context = stream_context_create(['http' => ['method' => 'POST', 'header' => "Content-Type: application/x-www-form-urlencoded\r\n", 'content' => $payload, 'timeout' => 10]]);
$response = file_get_contents($url, false, $context);
return json_decode((string) $response, true) ?: [];
```

## Модуль 5. Главная страница и Dashboard

Главная страница предназначена для предоставления менеджеру оперативной информации о состоянии туристических продаж и быстрых действий, связанных с Bitrix24. Необходимость dashboard обусловлена тем, что пользователь коммерческой CRM должен видеть не отдельные таблицы, а сводную картину: динамику продаж, наличие заявок, возможность отправить уведомление или создать задачу. Архитектура модуля включает `DashboardController`, представление `dashboard.php` и клиентский сценарий Chart.js.

Алгоритм работы состоит в обработке GET-запроса к корневому маршруту, получении профиля Bitrix24 методом `profile`, рендеринге HTML-страницы и построении графика средствами Chart.js. Пользовательский интерфейс содержит карточки, график, форму отправки системного уведомления и форму создания задачи. Взаимодействие с Bitrix24 выполняется через серверный REST-клиент, а также через JavaScript SDK, который подгоняет размер окна встраивания.

Входными данными являются запрос пользователя и сохраненный токен Bitrix24. Выходными данными являются HTML-страница и, при отправке форм, REST-запросы на создание уведомления или задачи. Бизнес-логика заключается в концентрации часто используемых операций на одной странице, что повышает операционную эффективность менеджера.

```php
return $this->view('dashboard', [
    'title' => 'Dashboard',
    'bitrixProfile' => (new BitrixApiClient())->call('profile'),
]);
```

## Модули 6–10. Меню, клиенты, туристические заявки, каталог туров и коммуникации

Данные модули образуют операционное ядро туристического приложения. Меню обеспечивает навигационную связность интерфейса; модуль клиентов хранит контактные данные туристов; модуль заявок описывает намерение клиента приобрести туристический продукт; каталог туров хранит коммерческие предложения; история коммуникаций фиксирует взаимодействия менеджеров с клиентами. Их разработка необходима для формирования замкнутого цикла продаж: регистрация клиента, выявление потребности, подбор тура, коммуникация и последующее сопровождение.

Архитектура реализована через общий абстрактный `ResourceController`. Такой выбор устраняет дублирование и фиксирует единый алгоритм для справочных модулей: получить список записей, отобразить форму и таблицу, принять POST-запрос, отфильтровать разрешенные поля и сохранить запись через репозиторий. Конкретные контроллеры клиентов, заявок, туров и коммуникаций определяют только имя таблицы, шаблон, путь перенаправления и допустимые поля. Это соответствует принципу открытости/закрытости: добавление нового справочника требует расширения, но не изменения общего алгоритма.

Пользовательский интерфейс представлен формами ввода и таблицами. Для клиентов входными данными являются ФИО, телефон и email; для заявок — идентификатор клиента, направление, бюджет и статус; для туров — название, страна и цена; для коммуникаций — клиент, канал и текст сообщения. Выходными данными являются HTML-таблицы и записи в MySQL. Прямое взаимодействие с Bitrix24 в этих модулях не обязательно, однако архитектура допускает расширение за счет методов `crm.contact.add`, `crm.deal.add` и `crm.timeline.comment.add`, если требуется синхронизация с CRM Bitrix24.

```php
abstract class ResourceController extends Controller
{
    protected string $table;
    protected string $template;
    protected string $redirectPath;
    protected array $allowedFields = [];

    public function store(array $request): string
    {
        $payload = array_intersect_key($request, array_flip($this->allowedFields));
        if ($payload !== []) {
            (new GenericRepository($this->container->get(PDO::class)))->create($this->table, $payload);
        }
        return $this->redirect($this->redirectPath);
    }
}
```

## Модули 11–14. Отчёты, уведомления и задачи Bitrix24

Аналитические и интеграционные модули предназначены для превращения локальной базы туристических операций в управленческий инструмент. Отчёты визуализируют накопленные данные, уведомления позволяют оперативно информировать сотрудников внутри Bitrix24, а интеграция с задачами обеспечивает перенос действий менеджера в стандартный механизм контроля исполнения. Необходимость этих модулей обусловлена коммерческой эксплуатацией: система должна не только хранить данные, но и поддерживать управленческие решения и контроль операций.

Архитектура уведомлений и задач строится на отдельных сервисах `NotificationService` и `TaskService`, которые используют `BitrixApiClient`. Алгоритм уведомления принимает идентификатор пользователя и текст сообщения, после чего вызывает REST-метод `im.notify.system.add` с параметрами `USER_ID` и `MESSAGE`. Алгоритм создания задачи принимает название, ответственного и описание, после чего вызывает `tasks.task.add` с объектом `fields`, содержащим `TITLE`, `RESPONSIBLE_ID` и `DESCRIPTION`. Отчеты отображаются отдельным контроллером и визуализируются Chart.js.

Входными данными являются параметры HTML-форм и сохраненный токен Bitrix24. Выходными данными являются REST-ответ Bitrix24, новая задача в портале или системное уведомление пользователю. Ошибки отсутствия токена возвращаются REST-клиентом как `NO_TOKEN`, что позволяет интерфейсу в дальнейшем показать диагностическое сообщение без падения приложения.

```php
return $this->bitrix->call('tasks.task.add', [
    'fields' => [
        'TITLE' => $title,
        'RESPONSIBLE_ID' => $responsibleId,
        'DESCRIPTION' => $description,
    ],
]);
```

## Модули 15–19. Настройки, логирование, обработка ошибок, тестирование и подготовка к установке

Финальные эксплуатационные модули обеспечивают пригодность приложения к внедрению. Настройки позволяют адаптировать поведение системы под конкретный портал. Логирование необходимо для диагностики инцидентов. Обработка ошибок предотвращает раскрытие технической информации пользователю. Тестирование подтверждает работоспособность ключевых классов. Подготовка к установке описывает действия администратора: настройку HTTPS-домена, заполнение `.env`, выполнение SQL-миграции, установку Composer-зависимостей и регистрацию URL в Bitrix24.

Архитектурно настройки реализованы контроллером `SettingsController`, который сохраняет конфигурацию в `storage/settings.json`. Глобальная обработка исключений подключается при загрузке контейнера. Тестовый сценарий проверяет наличие ключевых классов и тем самым подтверждает корректность автозагрузки. База данных подготавливается миграцией, что обеспечивает воспроизводимость окружения.

Результатом разработки является готовый минимально жизнеспособный каркас встроенного приложения Bitrix24, соответствующий требованиям PHP 8, MVC, ООП, PSR-подхода, MySQL, Bootstrap 5, Chart.js и REST API Bitrix24. Система может быть развернута как основа коммерческого решения и расширена специализированной доменной логикой туристического бизнеса.

```php
set_exception_handler([ExceptionHandler::class, 'render']);
```
