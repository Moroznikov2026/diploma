# Руководство по установке встроенного приложения FEON CRM в Bitrix24

Настоящее руководство описывает полный путь от подготовки компьютера разработчика до появления рабочего пункта FEON CRM в левом меню Bitrix24. Инструкция рассчитана на пользователя, который раньше не создавал Embedded Application. Описание основано на официальной модели Bitrix24 Local Applications, OAuth 2.0, BX24.js и механизме `placement.bind`: локальное приложение описывается внутри конкретного портала, получает доступ к REST API через OAuth 2.0, а элемент левого меню создается как виджет с `PLACEMENT=LEFT_MENU`, `HANDLER` и `TITLE`.

## 1. Что необходимо установить на компьютер

На компьютер разработчика необходимо установить PHP 8.1 или новее, Composer, MySQL 8 или MariaDB 10.6+, Git, веб-сервер Apache или Nginx, браузер Chrome либо Firefox, редактор кода и утилиту для HTTPS-туннеля. Это действие необходимо потому, что Bitrix24 вызывает приложение по публичному HTTPS-адресу, а приложение FEON CRM использует PHP backend, Composer-автозагрузку, MySQL-хранилище и JavaScript SDK BX24. Внутри Bitrix24 на данном этапе ничего еще не меняется; подготавливается только локальная среда. Из файлов проекта изменяются `.env`, будущая база данных и, при необходимости, конфигурация веб-сервера. Результатом должен стать компьютер, на котором из терминала выполняются команды `php -v`, `composer -V`, `mysql --version` и `git --version`.

## 2. Какой сервер использовать

Для разработки допускается использовать локальный PHP-сервер, Apache или Nginx. Для коммерческой эксплуатации следует использовать Linux-сервер с Nginx или Apache, PHP-FPM, MySQL и HTTPS-сертификатом. Локальный PHP-сервер подходит для первых тестов, но Bitrix24 должен получить доступ к приложению извне, поэтому локальный адрес `http://localhost:8000` необходимо опубликовать через HTTPS-туннель, например `ngrok`, `cloudflared tunnel` или аналогичный корпоративный шлюз. Внутри Bitrix24 в дальнейшем в настройки приложения будет записан именно публичный HTTPS URL, например `https://feon-crm.example.com/install.php`. Результатом этого шага является выбранная схема запуска: локально через туннель или на постоянном сервере.

## 3. Как настроить PHP

В файле `php.ini` необходимо включить расширения `pdo_mysql`, `openssl`, `mbstring`, `json`, `curl` и `fileinfo`, задать `date.timezone`, увеличить `memory_limit` минимум до `256M`, а для разработки включить отображение ошибок. Это требуется потому, что FEON CRM подключается к MySQL через PDO, обменивается данными с Bitrix24 по HTTPS, обрабатывает JSON-ответы REST API и использует Composer-пакеты. Внутри Bitrix24 на этом шаге ничего не создается. Изменяется системный файл `php.ini`, а в проекте позднее будет заполнен `.env`. После изменения конфигурации необходимо перезапустить веб-сервер или PHP-FPM и проверить `php -m`, чтобы убедиться, что расширения загружены.

## 4. Как установить Composer

Composer устанавливается с официального сайта getcomposer.org либо пакетным менеджером операционной системы. После установки необходимо выполнить `composer -V`. Composer нужен для PSR-4 автозагрузки классов `App\...` и установки библиотек, указанных в `composer.json`. Внутри Bitrix24 при этом ничего не изменяется. В проекте после выполнения `composer install` появятся каталог `vendor/` и файл `composer.lock`. Ожидаемый результат — команда `composer install` успешно создает зависимости, а команда `composer test` запускает smoke-тест.

## 5. Как создать локальный сервер

В каталоге проекта необходимо выполнить `php -S 127.0.0.1:8000 -t public`. Данная команда запускает web-root `public`, поэтому пользователю не будут доступны исходные файлы `src`, `.env` и SQL-миграции. Затем следует открыть `http://127.0.0.1:8000/health`. Если используется Bitrix24, локальный сервер необходимо опубликовать: например, `ngrok http 8000`, после чего будет получен HTTPS-адрес. Внутри Bitrix24 этот адрес позднее будет указан в URL установки, URL обработчика и callback URL. Результатом должен быть открывающийся endpoint `/health` и публичный HTTPS URL.

## 6. Как зарегистрировать Embedded Application в Bitrix24

Необходимо войти в портал Bitrix24 под пользователем с правами администратора. Далее в левом меню следует открыть `Приложения`, затем перейти в раздел `Разработчикам` или `Developer resources`, выбрать сценарий создания локального приложения и тип `Local application`. Если интерфейс портала отличается, следует искать раздел локальных интеграций или ресурсов разработчика. Это действие необходимо, потому что локальное приложение создается и описывается непосредственно в конкретном портале Bitrix24, а администратор определяет его название, права и URL. Внутри Bitrix24 создается карточка приложения. В файлах проекта пока ничего не изменяется. Результатом должна стать форма настройки локального приложения FEON CRM.

## 7. Как получить Client ID

После сохранения или создания локального приложения Bitrix24 отображает `Client ID`. Его необходимо скопировать в `.env` в параметр `BITRIX24_CLIENT_ID`. Client ID является публичным идентификатором приложения в OAuth 2.0 и используется Bitrix24 для понимания, какое приложение запрашивает авторизацию. Внутри портала значение генерируется автоматически для конкретной локальной интеграции. Результатом должен быть заполненный параметр `.env`: `BITRIX24_CLIENT_ID=local.xxxxxxxxx`.

## 8. Как получить Client Secret

В той же форме локального приложения Bitrix24 отображает `Client Secret`. Его необходимо скопировать в `.env` в параметр `BITRIX24_CLIENT_SECRET` и не публиковать в репозитории. Client Secret является закрытым ключом OAuth-приложения и подтверждает, что запрос токена выполняет именно backend FEON CRM. Внутри Bitrix24 секрет привязан к локальному приложению. Результатом должен быть заполненный локальный `.env`; файл `.env.example` остается шаблоном и не содержит реальных секретов.

## 9. Как настроить OAuth

В карточке приложения необходимо указать redirect URI, URL установки и права доступа. Для текущего проекта redirect/install URL должен указывать на `https://ваш-домен/install.php`. OAuth нужен потому, что Bitrix24 передает приложению код или установочные токены, а приложение сохраняет `AUTH_ID`, `REFRESH_ID`, `DOMAIN`, `member_id` и `client_endpoint`. Внутри Bitrix24 OAuth связывает портал, пользователя-администратора и локальное приложение. Из файлов проекта должен быть заполнен `.env`, особенно `APP_URL`, `BITRIX24_CLIENT_ID`, `BITRIX24_CLIENT_SECRET` и `BITRIX24_REDIRECT_URI`. Ожидаемый результат — при установке Bitrix24 вызывает `public/install.php`, а файл `storage/tokens.json` получает данные авторизации.

## 10. Как подключить приложение к порталу Bitrix24

В настройках локального приложения необходимо указать публичные URL: URL установки `https://ваш-домен/install.php`, URL приложения или обработчика `https://ваш-домен/placement.php`, redirect URI `https://ваш-домен/install.php`. Затем требуется выбрать права доступа, минимально `basic`, `im` и `task`, поскольку приложение использует методы `profile`, `im.notify.system.add`, `tasks.task.add` и `placement.bind`. Если в дальнейшем будет включена синхронизация CRM, дополнительно потребуется право `crm`. Внутри Bitrix24 сохраняется конфигурация локального приложения и перечень scope. В проекте изменяется `.env`. Результатом является готовая карточка приложения, которую можно установить.

## 11. Как настроить Callback URL

Callback URL должен быть публичным HTTPS-адресом `install.php`. В поле callback или redirect URI в Bitrix24 необходимо ввести абсолютный URL, например `https://feon-crm.example.com/install.php`. Это необходимо, чтобы Bitrix24 знал, куда вернуть пользователя и куда передать параметры установки. Внутри Bitrix24 этот URL сохраняется в OAuth-настройках приложения. В проекте соответствующее значение указывается в `.env` как `BITRIX24_REDIRECT_URI`. Результат проверяется установкой: если callback указан правильно, после установки пользователь будет перенаправлен в FEON CRM.

## 12. Как развернуть структуру проекта

На сервере необходимо создать каталог приложения, например `/var/www/feon-crm`, выполнить `git clone` или скопировать файлы проекта, затем выполнить `composer install --no-dev --optimize-autoloader` для production или `composer install` для разработки. Web-root сервера должен указывать только на `/var/www/feon-crm/public`. Это необходимо для безопасности: файлы `.env`, `src`, `database` и `storage` не должны быть доступны напрямую из браузера. Внутри Bitrix24 ничего не меняется; меняется серверная файловая система. Результатом должна быть структура, в которой публично открываются только `index.php`, `install.php`, `placement.php` и статические assets.

## 13. Куда помещается каждый файл проекта

Файлы `public/index.php`, `public/install.php`, `public/placement.php` и `public/assets` размещаются внутри web-root. Каталог `src` содержит PHP-код backend и размещается уровнем выше web-root. Каталог `database/migrations` содержит SQL-миграции. Каталог `storage` хранит токены, настройки и логи, поэтому должен быть доступен на запись пользователю веб-сервера. Файл `.env` размещается в корне проекта и не должен быть доступен по HTTP. Это разделение необходимо для защиты секретов. Внутри Bitrix24 результатом будет корректная загрузка публичных URL, а на сервере — отсутствие прямого доступа к исходным и конфигурационным файлам.

## 14. Как подключить JavaScript SDK BX24

В шаблоне `src/View/layout.php` подключается `<script src="https://api.bitrix24.com/api/v1/"></script>`, а в `public/assets/js/app.js` вызывается `BX24.init`. Это необходимо, потому что BX24.js является SDK для приложений, встроенных в интерфейс Bitrix24, и после инициализации получает данные от родительского фрейма. Внутри Bitrix24 при открытии placement приложение загружается во фрейме, а SDK запрашивает контекст у родительского интерфейса. Изменяемые файлы — `src/View/layout.php` и `public/assets/js/app.js`. Результатом является отсутствие ошибок `BX24 is not defined` и корректная подгонка окна через `BX24.fitWindow()`.

## 15. Как подключить PHP backend

PHP backend подключается через `public/index.php`, который загружает `vendor/autoload.php`, инициализирует контейнер и маршрутизатор. Веб-сервер должен направлять все основные запросы на этот файл или обслуживать прямые маршруты, если используется встроенный PHP-сервер. Это необходимо для MVC-архитектуры и единого входа в приложение. Внутри Bitrix24 backend вызывается при открытии `placement.php`, а затем пользователь перенаправляется в основное приложение. Изменяемые файлы — конфигурация Nginx/Apache и `.env`. Результатом является открытие страниц `/`, `/clients`, `/requests`, `/tours`, `/communications`, `/reports` и `/settings`.

## 16. Как подключить базу данных

В MySQL необходимо создать базу, например `CREATE DATABASE feon_crm CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;`, создать пользователя и выдать права. Затем в `.env` указываются `DB_HOST`, `DB_PORT`, `DB_DATABASE`, `DB_USERNAME`, `DB_PASSWORD`. Это необходимо, потому что модули клиентов, заявок, туров и коммуникаций сохраняют данные локально. Внутри Bitrix24 база данных напрямую не создается; портал передает данные через REST и iframe, а FEON CRM хранит собственные сущности на сервере. Результатом должно быть успешное подключение PDO без ошибки `SQLSTATE[HY000]`.

## 17. Как выполнить SQL миграции

Необходимо выполнить команду `mysql -u feon_user -p feon_crm < database/migrations/001_create_tables.sql`. Миграция создает таблицы `clients`, `travel_requests`, `tours` и `communications`. Это необходимо, чтобы формы приложения могли сохранять и читать данные. Внутри Bitrix24 на этом шаге ничего не меняется. Изменяется структура MySQL. Результатом должна быть команда `SHOW TABLES;`, возвращающая четыре таблицы приложения.

## 18. Как протестировать подключение к REST API

Сначала нужно установить приложение в Bitrix24, чтобы появился `storage/tokens.json`. Затем следует открыть главную страницу FEON CRM. Контроллер Dashboard вызывает REST-метод `profile`, а уведомления и задачи используют `im.notify.system.add` и `tasks.task.add`. Для ручной проверки можно отправить форму уведомления на главной странице, указав ID пользователя Bitrix24. Это действие необходимо, чтобы убедиться, что токен сохранен, endpoint корректен, а scope достаточен. Внутри Bitrix24 должен появиться профильный REST-ответ, уведомление или задача. Если результат `NO_TOKEN`, приложение не установлено или файл токена не создан.

## 19. Как установить приложение в Bitrix24

В карточке локального приложения необходимо нажать кнопку установки или открыть сформированную ссылку установки из интерфейса Bitrix24. Установку должен выполнять администратор портала. Bitrix24 откроет URL установки FEON CRM и передаст параметры авторизации. `public/install.php` передаст параметры в `BitrixAuthService`, а тот сохранит токены через `TokenStorageService`. Внутри Bitrix24 приложение получает статус установленного. В проекте изменяется `storage/tokens.json`. Результатом является редирект на главную страницу приложения и наличие сохраненных данных авторизации.

## 20. Как добавить пункт приложения в левое меню Bitrix24

После установки нужно открыть `FEON CRM → Настройки` и нажать кнопку `Добавить в левое меню`. Форма отправляет POST-запрос на `/placements/left-menu/bind`, а backend вызывает REST-метод `placement.bind` с параметрами `PLACEMENT=LEFT_MENU`, `HANDLER=https://ваш-домен/placement.php` и `TITLE=FEON CRM`. Это необходимо, чтобы приложение отображалось не только в списке приложений, но и как отдельный пункт навигации Bitrix24. Внутри Bitrix24 создается виджет главного левого меню. Файлы проекта не изменяются; меняется конфигурация портала. Результатом должен стать пункт `FEON CRM` в левом меню Bitrix24.

## 21. Как обновлять приложение после изменения кода

После изменения кода необходимо выполнить `git pull` или загрузить новые файлы на сервер, затем выполнить `composer install --no-dev --optimize-autoloader`, очистить OPcache или перезапустить PHP-FPM, выполнить новые миграции и открыть `/health`. Это необходимо, чтобы сервер использовал новую версию классов и схемы базы. Внутри Bitrix24 обычно ничего менять не требуется, если URL приложения не изменились. Изменяются файлы проекта и, при наличии миграций, структура MySQL. Результатом является обновленное приложение, доступное по прежнему пункту `FEON CRM`.

## 22. Как публиковать новую версию

Для публикации новой версии следует создать Git-тег, подготовить changelog, выполнить деплой на staging, проверить установку в тестовом портале Bitrix24, затем выполнить деплой на production. Если изменились URL, scope или placement, необходимо обновить настройки локального приложения в Bitrix24. Это действие необходимо для контролируемой эксплуатации и возможности отката. Внутри Bitrix24 могут измениться права приложения или обработчики. В проекте изменяются исходные файлы, миграции и документация. Результатом является версия FEON CRM, доступная пользователям без ручного вмешательства в их рабочий процесс.

## 23. Как удалить приложение

Для удаления необходимо сначала открыть настройки FEON CRM и нажать удаление пункта меню, что вызовет `placement.unbind` для `LEFT_MENU`. Затем в Bitrix24 перейти в `Приложения → установленные приложения`, выбрать FEON CRM и нажать удаление. На сервере можно удалить токены `storage/tokens.json` и при необходимости удалить базу MySQL. Это необходимо, чтобы в портале не оставались неработающие пункты меню и активные токены. Внутри Bitrix24 удаляется установка приложения и placement. Результатом является отсутствие FEON CRM в меню и списке установленных приложений.

## 24. Как выполнять отладку

Для отладки необходимо включить `APP_DEBUG=true` в `.env`, смотреть логи веб-сервера, PHP-FPM и `storage/logs/app.log`, проверять наличие `storage/tokens.json`, выполнять `php tests/run.php` и синтаксическую проверку `find src public tests -name '*.php' -print0 | xargs -0 -n1 php -l`. В Bitrix24 следует проверять, что приложение открыто именно внутри портала и что домен в токене совпадает с текущим порталом. Это необходимо для локализации ошибок между frontend, backend, OAuth, REST и базой данных. Результатом должна быть воспроизводимая причина ошибки и исправленный файл проекта или настройка портала.

## 25. Как исправлять типичные ошибки

Если появляется ошибка `BX24 is not defined`, необходимо проверить подключение `https://api.bitrix24.com/api/v1/` и открытие приложения внутри Bitrix24, а не напрямую в браузере. Если REST возвращает `NO_TOKEN`, нужно переустановить приложение и проверить права на запись в `storage`. Если Bitrix24 не открывает приложение, нужно проверить HTTPS URL, callback URL и доступность `/install.php` и `/placement.php` из интернета. Если формы не сохраняют данные, нужно проверить `.env`, PDO-драйвер и выполнение SQL-миграции. Если пункт меню не появляется, нужно проверить scope `basic`, установленное приложение и ответ метода `placement.bind`. Результатом исправления должно быть полностью работоспособное встроенное приложение FEON CRM внутри Bitrix24: оно открывается из левого меню, показывает интерфейс, сохраняет локальные данные, вызывает REST API и создает уведомления или задачи.

## Контрольный чек-лист результата

После завершения установки должны выполняться следующие условия. Домен приложения доступен по HTTPS. В Bitrix24 создано локальное приложение FEON CRM. В `.env` заполнены параметры приложения, базы данных и OAuth. SQL-миграции выполнены. После установки существует `storage/tokens.json`. Главная страница открывается внутри Bitrix24. JavaScript SDK инициализируется без ошибок. REST-метод `profile` возвращает данные текущего пользователя. Метод `im.notify.system.add` отправляет уведомление. Метод `tasks.task.add` создает задачу. Метод `placement.bind` создает пункт `FEON CRM` в левом меню.

## Использованные официальные сведения Bitrix24

Официальная документация Bitrix24 описывает локальные приложения как приложения, которые создаются внутри конкретного портала и получают доступ к REST API через OAuth 2.0. Документация OAuth поясняет обмен authorization code, `client_id` и `client_secret`. Документация installation callback указывает, что `client_id` и `client_secret` берутся из формы локального приложения, а redirect URI задается в настройках приложения. Документация BX24.js указывает, что SDK применяется для встроенных приложений и подключается через `https://api.bitrix24.com/api/v1/`. Документация `BX24.init` поясняет, что инициализация получает данные от родительского фрейма. Документация виджетов левого меню описывает `LEFT_MENU`, а справка по приложениям с виджетами описывает параметры `PLACEMENT`, `HANDLER` и `TITLE` для `placement.bind`.
